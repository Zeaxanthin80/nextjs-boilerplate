1. Quick Summary - Key Definitions

Essential Terms

Term Definition
PR (Pull Request) A GitHub feature to propose and review code changes before merging into main branch
npm run dev Basic Next.js local development server using local environment
vercel dev Enhanced development server that mimics Vercel's production environment
Preview Deployment Temporary deployment of your changes for testing (automatically created from PRs)
Production Deployment Live website accessible to users (deployed from main branch)

Additional Useful Terms

  • Branch: Separate line of development in Git (like a copy of your code to work on)
  • Commit: Snapshot of your code changes with a descriptive message
  • Merge: Combining changes from one branch into another (usually into main)
  • Environment Variables: Configuration values (API keys, database URLs) stored securely
  • Serverless Functions: Code that runs on-demand without managing servers (in /api folder)
  • Edge Runtime: Vercel's global network that runs your code closer to users
  • Hot Reload: Automatic page refresh when you save code changes
  • CI/CD: Continuous Integration/Continuous Deployment (automated testing and deployment)
  • Rollback: Reverting to a previous version if something breaks
  • Staging: Testing environment that mimics production

8. Command Reference

Git Commands

# Branch management
git checkout -b feature-name
# Create and switch to new branch
git checkout main
# Switch to main branch
git branch
# List all branches
git branch -d branch-name
# Delete branch

# Changes management
git status
# Check current status
git add .
# Stage all changes
git add filename
# Stage specific file
git commit -m "message"
# Commit changes
git push origin branch-name
# Push branch to GitHub
git pull origin main
# Pull latest changes from main

# View history
git log --oneline
# View commit history
git diff
# View unstaged changes

Vercel CLI Commands

# Development
vercel dev
# Start development server
vercel --prod=false
# Create preview deployment
vercel --prod
# Deploy to production
vercel link
# Connect local project to Vercel

# Environment management
vercel env pull .env.local
# Download production env vars
vercel env ls
# List environment variables
vercel env add
# Add new environment variable

# Project management
vercel ls
# List all deployments
vercel inspect <url>
# Get deployment details
vercel promote <url>
# Promote preview to production
vercel rollback <url>
# Rollback to previous deployment

# Logs and debugging
vercel logs <url>
# View deployment logs
vercel whoami
# Check current user
vercel --version
# Check CLI version

NPM Commands

npm run dev
# Start basic Next.js dev server
npm run build
# Build for production
npm run start
# Start production server locally
npm install
# Install dependencies
npm install package-name
# Install specific package

9. Troubleshooting

Common Issues

1. "vercel command not found"

# Solution: Install globally
npm i -g vercel

# Or use npx
npx vercel

2. "Project not linked"

# Solution: Link your project
vercel link

# Follow the prompts to connect to your existing Vercel project

3. "Environment variables not working"

# Solution: Pull from production
vercel env pull .env.local

# Then restart your dev server

Updated Recommendation: Use npm run dev for Team Members

Solution: Based on Vercel's official documentation, team members should use Next.js's built-in development server:

# Recommended approach for Norma and Lily:
npm run dev
# This provides all the features needed for Next.js development
# No authentication required, works immediately

Why this works better:

  • npm run dev includes hot reload, API routes, and all Next.js features
  • No authentication or login prompts
  • Officially recommended by Vercel for Next.js projects
  • Perfect for frontend/UI work (Norma) and feature development (Lily)

When to use vercel dev: Only Jose needs this for infrastructure work, deployment testing, and accessing production environment variables.

5. "Should I use vercel dev or npm run dev?"

Answer: For most team members, use npm run dev

Explanation:

# For Norma (Frontend/UI) and Lily (Features/Integration):
npm run dev
# Perfect for: React components, styling, API integration, features
# Includes: Hot reload, API routes, Next.js routing, fast refresh

# For Jose (Infrastructure/Lead) only when needed:
vercel dev
# Needed for: Production environment testing, deployment preparation

Vercel's Official Recommendation: "If you're using a framework and your framework's Development Command already provides all the features you need, we do not recommend using vercel dev."

4. "Git push rejected"

# Solution: Pull latest changes first
git pull origin main

# Then push again
git push origin your-branch-name

6. "Preview deployment failed"

Check the Vercel dashboard for build logs:

# Or check logs via CLI (Jose only)
vercel logs <deployment-url>

10. Best Practices

1. Branch Naming

Use descriptive branch names:

git checkout -b feature/contact-form
git checkout -b fix/navbar-mobile
git checkout -b update/dependencies

2. Commit Messages

Write clear commit messages:

git commit -m "Add contact form with validation"
git commit -m "Fix responsive navbar on mobile"
git commit -m "Update dependencies to latest versions"

3. Environment Variables

  • ‚ùåNever commit sensitive data to Git
  • ‚úÖUse vercel env pull to sync production variables
  • ‚úÖKeep .env.local in .gitignore

4. Testing Strategy

  1. Local testing: Use vercel dev for production-like environment
  2. Preview testing: Create PR for team review
  3. Production monitoring: Watch for errors after deployment

5. Deployment Strategy

  • Small changes: Direct merge to main
  • Large features: Use feature branches and PRs
  • Urgent fixes: Use vercel --prod for emergency deployments

6. Team Collaboration

  • ‚úÖAlways create PRs for code review
  • ‚úÖUse descriptive PR titles and descriptions
  • ‚úÖTest preview deployments before merging
  • ‚úÖCommunicate with team about breaking changes
