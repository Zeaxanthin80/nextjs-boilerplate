2. Direct Collaboration vs Forking

‚úÖ Direct Collaboration (RECOMMENDED)

Your Current Setup

How It Works:
# Team members work directly in your repository
git clone https://github.com/jose/nextjs-boilerplate.git
git checkout -b feature/contact-form
# Work on feature...
git push origin feature/contact-form
# Create PR directly in main repo
‚úÖ PROS
  • Simple setup - 4 steps to get started
  • No syncing overhead - Always up to date
  • Clear ownership - One main repository
  • Easy branch management - All branches in one place
  • Streamlined PRs - Direct pull requests
  • Perfect for small teams - 3-5 trusted developers
‚ùå CONS
  • Requires trust - Team members need write access
  • Shared responsibility - Need branch protection
  • Less isolation - Branches visible to all

‚ùå Fork Workflow (NOT RECOMMENDED)

Alternative Approach

How It Works:
# Each team member forks your repository
git clone https://github.com/norma/nextjs-boilerplate.git
git remote add upstream https://github.com/jose/nextjs-boilerplate.git
git checkout -b feature/contact-form
# Work on feature...
git push origin feature/contact-form
# Create PR from their fork to your main repo

# Daily sync overhead:
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
‚úÖ PROS
  • Complete isolation - Each developer owns their fork
  • No write access needed - Can contribute without permissions
  • Great for open source - Standard for public projects
‚ùå CONS
  • Complex setup - 8+ steps to get started
  • Constant syncing - Must keep forks updated
  • Sync conflicts - Easy to get out of sync
  • Confusing for beginners - Multiple remotes
  • Overkill for small teams - Unnecessary complexity
  • Slower development - More overhead

üéØ When to Use Each Approach

Use Direct Collaboration For:

  • ‚úÖ Small trusted teams (2-10 people)
  • ‚úÖ Private repositories
  • ‚úÖ Capstone projects
  • ‚úÖ Startup teams
  • ‚úÖ Internal company projects
  • ‚úÖ Learning environments

Use Fork Workflow For:

  • üåê Open source projects
  • üë• Large teams (50+ contributors)
  • üîí High security requirements
  • üìù Public repositories with external contributors
  • üè¢ Cross-organization collaboration

3. Public vs Private Repositories

üåê Public Repository

What Anyone Can Do:

  • ‚úÖ View all code - Repository is publicly visible
  • ‚úÖ Clone the repository - Download a copy
  • ‚úÖ Fork the repository - Create their own copy
  • ‚úÖ View commit history - See all changes
  • ‚úÖ Open issues - Report bugs or suggestions

What They CANNOT Do:

  • ‚ùå Push to your repository - No write access
  • ‚ùå Create branches - In your repository
  • ‚ùå Merge pull requests - No admin access
  • ‚ùå Change settings - Repository configuration

For Your Team to Collaborate:

‚ö†Ô∏è Must Invite as Collaborators

Even with a public repo, Norma and Lily need collaborator access to push branches and work directly in your repository.

üîí Private Repository

What Others Can Do:

  • ‚ùå Nothing - Repository is completely hidden
  • ‚ùå Cannot view code - No visibility
  • ‚ùå Cannot clone - No access
  • ‚ùå Cannot see it exists - Completely private

For Your Team to Collaborate:

‚úÖ Must Invite as Collaborators

Team members need explicit invitation and permissions to access private repositories.

‚úÖ Good vs ‚ùå Bad Branch Names

‚úÖ Good Branch Names

  • fix/mobile-navbar-collapse - Specific issue

  • docs/api-authentication-guide - Clear documentation scope
  • refactor/database-query-optimization - Specific improvement

‚ùå Bad Branch Names

  • feature/stuff - Too vague
  • fix/bug - No specificity
  • norma-work - Personal but unclear
  • temp-branch - Non-descriptive
  • asdf - Meaningless
